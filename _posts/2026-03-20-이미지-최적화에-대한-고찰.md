> 브라우저에 이미지를 불러오는 모든 고민의 흔적을 정리한 글이다.



## 브라우저에서 띄우는 이미지 대표 형식



- jpg(jpeg)
- png
- Webp
- AVIF
- svg



대표적으로 5가지가 존재한다. 각 이미지 형식은 압축하는 방식(알고리즘)에 따라서 구분해놨다고 보면된다.

300x300 픽셀의 크기를 가진 이미지가 있다고 했을때 총 9만개나되는 픽셀이 있고 각 픽셀은 약 3byte(4byte)정도의 크기를 필요로 하니 총 27만개의 메모리(트랜지스터)가 필요하다. 이를 그대로 저장하게 되면 기기의 메모리 점유율을 비약적으로 높인다. 이는 곧 응답 속도 저하로 이어질 수 있다.(사용자 경험 저하)



> 아래서 정리한 리스트는 실제 이미지 압축 방식 알고리즘을 정확히 모두 설명하지 않는다. 자세한 압축 알고리즘을 알고싶으면 검색을 추천한다.



### 1. Jpg(jpeg)



대표적인 손실압축 방식이다. 
말그대로 사용자가 이미지를 봤을때 구분이 안되는 영역을 내부 알고리즘으로 압축해 이미지를 저장하는 방식이다.
손실압축이라 손실율을 이미지를 압축할때 설정할 수 있고 원본 이미지와 다르게 화질이 좋지 않게 나올 수 있다. 



### 2. png



대표적인 무손실 압축이다 원본 이미지와 거의 동일한 해상도로 이미지를 저장하는 방식이다. 그렇기 때문에 용량이 크다. 



하지만 png라고 픽셀 데이터를 그대로 저장하지는 않는다. 내부적으로 주변 픽셀들의 색을 비교해 각 픽셀의 데이터를 그대로 저장하는게 아닌 기준값에서 얼마나 차이가 났는지만 저장하는 방식으로 압축을 해 해상도만큼의 메모리가 필요하지는 않는다. 

=> **그래서 같은 해상도의 이미지여도 비슷한 색을 사용하는 픽셀이 많은 이미지가 용량이 더욱 작다.**



### 3. Webp



앞의 jpg와 png의 단점을 더욱 개선한 압축 방식의 이미지 형식이라고 볼 수 있다.
그래서 jpg, png의 장점을 모두 흡수한 압축 방식이다. (용량도 작고 화질도 보존 가능)



하지만 지금은 단점이 아닐 수 있지만 예전 브라우저인 IE등은 이 이미지 형식을 지원하지 않는다. 



### 4. AVIF



Webp 방식에서 한층더 발전된 압축 알고리즘을 가진 이미지 형식이다. 그러므로 기존 이미지를 AVIF로 인코딩 하는데 Webp보다 더욱 시간이 오래걸린다. 
고해상도의 최신 브라우저에서 서비스하는 경우 사용하면 좋을 이미지라고 생각이든다.



범용성 측면에서는 아직은 Webp가 AVIF보다 훨씬 좋다. 안정적으로 이미지를 서빙하려면 Webp를 추천한다고 한다.



## 이미지의 형식의 특징을 개발자가 알아야할까?



> 이미지 형식을 아는게 왜 개발자에게 중요할까? 특히 프론트엔드 개발자들한테



위 형식들의 특징들을 보면 각각의 트레이드 오프 관계가 있다. 무조건적으로 모든 서비스에 이런 이미지 형식을 써야돼! 라는게 없다는걸 느낄 수 있다.



대표적으로 사용자 경험을 좋게 하려면 빠르게 페이지를 눈으로 확인하고 상호작용이 가능해야한다.

이 때 이미지같은 용량이 큰 데이터를 서버에서 받아와서 렌더링 시켜주려면 이미지의 압축률 즉 용량이 작을수록 유리하다. 그리고 이미지의 화질도 보장도 동시에 보장이 되어야한다. 

jpg를 쓰자니 용량이 작아 LCP에는 좋지만 이미지 화질을 보장하지 못한다. 그렇다고 png를 쓰자니 이미지 화질을 보장하지만 용량이 너무커 LCP에는 좋지 못하다.

거기에대한 대안으로 Webp를 쓰면 좋지만 아주 예전 브라우저를 사용하는 사용자에게는 해당 이미지에 대한 디코더가 없기 때문에 이미지가 보이지 않는다.



**즉 결론은 각 이미지 형식의 트레이트 오프 관계를 이해하고 비교해서 서비스 성격에 맞는 서비스를 만드는게 필요하다.**



## 실제 서비스에서 이미지를 불러오기전에 알면 좋을 이슈



> 서비스에 이미지를 그대로 사용했을때 생기는 이슈에 대해 정리했다.



### DPR 이슈로 인한 화질 저하 이슈



![img](https://velog.velcdn.com/images%2Fvnfdusdl%2Fpost%2F0d7f13b4-1830-4922-b9f6-bd657e589575%2Fimage.png)

이미지 출처 : https://velog.io/@vnfdusdl/DPRDevice-pixel-ratio%EC%9D%98-%EC%9D%B4%ED%95%B4

DPR이란 간단하게 설명하면 Device Per Ratio라는 의미로 실제로 기기에 보이는 논리적인 픽셀인 css의 1px을 표현하려고 물리적인 픽셀이 얼마나 사용됐는지의 비율이다.
DPR값이 1이면 css 1px을 표현하려면 하나의 픽셀이 대응돼고 DPR값이 2이면 css 1px을 표현하기 위해 물리적인 픽셀 2x2 즉 4개의 칸이 모여 하나의 px을 만들어낸다고 볼 수 있다.



![DPR이 1인 환경과 DPR이 2인 환경에서의 구글 로고](https://cdn.jsdelivr.net/gh/jaehyeon48/jaehyeon48.github.io@master/assets/images/htmlcss/diff-between-canvas-size/google_logo_in_different_dpr.png)

이미지 출처 : https://jaehyeon48.github.io/htmlcss/resolve-blurry-canvas-in-hi-resolution/#:~:text=%EA%B8%B0%EA%B8%B0%20%ED%94%BD%EC%85%80%20%EB%B9%84%EC%9C%A8%2C%20DPR

위 이미지를 png 형식으로 저장해서 브라우저에서 불러와서 렌더링해보면 화질이 다르게 나온다. 

DPR이 높은 기기에서 더 촘촘히 이미지를 표현하니 더 화질이 좋아야하는거 아니야? 라고 생각할 수 있는데 화면에 렌더링 되어야하는 크기와 이미지의 해상도를 비교해보면 이유를 알 수 있다.



예를들어 화면상에 200x600 크기의 이미지가 그려져야하고 이미지의 크기도 200x600이라면 DPR이 1인 기기에서는 왼쪽처럼 하나의 픽셀에 하나의 이미지의 픽셀이 들어가니 정상적으로 나오게 된다.

하지만 DPR이 2인 기기에서는 이미지 하나의 픽셀 정보를 가지고 2x2인 4개의 칸을 채워야한다. 이 때 보간법으로 주변 색을 섞어 4개의 칸을 칠해준다. 이 때문에 오른쪽처럼 실제로 렌더링되는 이미지는 조금 뿌옇게 나오게 된다.



이 경우 해결방법은 간단하다. 200x600 이미지를 받아오는게 아닌 x2한 값인 400x1200 해상도를 가진 이미지를 불러와서 띄우게 되면 선명하게 이미지를 확인할 수 있다.
하지만 반대로 더 큰 해상도를 필요로 하니 더욱 큰 용량의 이미지를 불러오게 되는 단점도 있다. 



참고로 DPR은 하나의 기기에서 듀얼 모니터로 연결했을때 각 모니터의 DPR에 따라 다르게 보인다. 



## 실제 서비스에서 이미지 사용하기



> 실제 서비스에서 이미지는 먼저 서비스에 올라와있는 정적 이미지를 활용하는 방법과 사용자가 동적으로 올리는 이미지를 띄워주는 방식 두 가지가 존재한다. 
>
> 이 부분에서는 실 서비스에서 이미지를 어떻게 최적화해서 넣어줘야 하는지에 대한 고민을 정리했다.



### 정적 이미지 최적화



서비스에 누가 들어가던 항상 나오는 이미지를 정적 이미지라고 한다. 

정적 이미지를 넣을꺼니 처음부터 이미지를 우리 서비스에 맞게 장단점을 따져 맞는 포맷으로 변환해 저장하는걸 추천한다. 보통 디자이너가 이미지 jpg, png 형태로 전달해주기 때문에 자체적으로 WebP로 변환해서 저장해서 불러오도록 하는편이다.

그리고 앞에 DPR 이슈에 대응하기 위해서 원본 이미지x1, 이미지x2, 이미지x3 세가지의 해상도를 가진 이미지를 별도로 저장한다.

이렇게 세가지 구분해서 저장하는 이유는 사용자가 보는 기기의 DPR에 따라 구분해서 맞는 이미지를 브라우저가 받아올 수 있도록 하기 위해서다.



```html
<picture>
  <source 
    media="(max-width: 767px)" 
    srcset="mobile-image-3x.webp 3x, mobile-image-2x.webp 2x, mobile-image-1x.webp 1x" 
    type="image/webp">
  <source srcset="image-3x.webp 3x, image-2x.webp 2x, image-1x.webp 1x" type="image/webp">
  <img src="image-1x.png" alt="설명" style="width: 300px;">
</picture>
```

이렇게 사용자의 기기의 DPR에 따라 해상도를 띄워주는 이미지를 불러오도록 해주면 된다. 브라우저는 똑똑하게 현재 사용자의 기기의 DPR에 해당하는 이미지만 요청하도록 동작한다.

그리고 마지막엔 img 태그를 넣어 혹시 사용자의 브라우저가 webp 디코더가 없는 경우 보여줄 이미지를 png 포맷으로 보여주도록 작성해주면 된다.

(참고로 DPR이 2.3 이면 브라우저는 화질의 보장을 위해 최소값보다 큰 x3 이미지를 요청한다.)



데스크탑과 모바일에 서로 다른 이미지를 넣고 싶은 경우는 media 속성을 사용해서 제한해주면 된다. picture 내에 태그들은 위에서부터 읽기때문에 767px보다 작은 모바일 기기에서는 가장 상단의 source 태그가 수행된다.



CSS를 통해 이미지를 넣을 수도 있다. 

```css
background-image: image-set(
    url("image-1x.webp") 1x,
    url("image-2x.webp") 2x,
    url("image-3x.webp") 3x
  );
```

배경 이미지를 넣을 경우 image-set를 이용해 기기의 DPR에 맞게 이미지를 불러올 수 있다.



### 동적 이미지 최적화



사용자가 우리 서비스를 통해 이미지를 넣는 경우를 의미한다. 
Ex) 커뮤니티 게시글, 프로필 사진 변경 등등



사용자가 첨부한 이미지는 서버에 저장되니 너무 큰 이미지를 저장하면 우리 서버가 부담이 될 수 있다.

여기에 대응할 수 있는 방법은 여러가지가 있을 수 있다.

- 사용자에게 이미지 업로드 용량 제한
- 사용자가 업로드한 이미지 서버에 보내기전에 클라에서 이미지 변환 작업
- 서버에서 이미지 저장하기 전에 이미지 변환 작업

서비스의 상황이나 팀에 따라서 다르게 처리하면 좋을꺼 같다. 



우리 서비스에는 두 번째 방법을 차용하고 있다. DPR 이슈도 있기 때문에 최대한 대응하기 위해 사용자가 업로드한 이미지의 해상도가 매우 크다면 서비스에 최대 너비에 두 배까지 줄이고 webp로 전환해서 업로드 하는 전략으로 처리하고 있다. (물론 최대 너비보다 작은 이미지를 업로드 하는 경우 너비를 조정하지 않는다.)
